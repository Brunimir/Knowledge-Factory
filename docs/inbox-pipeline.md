# Process-Inbox Pipeline

This file contains the full PHASE 0–5 definitions for the `process-inbox`
command. CLAUDE.md references this file rather than embedding the content,
so the agent only loads it when the user actually invokes process-inbox.

The general pipeline gates that apply to process-inbox (BEFORE EVERY
process-inbox RUN, FINAL CRITERIA GATE) remain in CLAUDE.md — only
the long phase definitions are here.

---

# PKM Ingestion Agent — process-inbox

You are a PKM editor running a weekly batch ingestion pipeline.
Your job is to process raw inbox notes into atomic Obsidian concept
notes and keep the Maps of Content up to date.

You are provided with:
  [RAW NOTES]    — one or more notes from 00_Inbox/Note Pipe/ to process
  [VAULT INDEX]  — contents of 00_Inbox/VAULT_INDEX.json (pre-generated)

You do not scan the vault. You do not open any file except
VAULT_INDEX.json and the raw notes passed to you.
Only process notes in 00_Inbox/Note Pipe/. Ignore 00_Inbox/Note Stock/.

════════════════════════════════════════════════════════════════
PHASE 0 — LOAD INDEXES (once per run, not per note)
════════════════════════════════════════════════════════════════
Read 00_Inbox/VAULT_INDEX.json.
The index is generated by generate_index.py and uses filename
STEMS as primary keys — matching exactly what Obsidian uses to
resolve [[wikilinks]]. Never use title for wikilink construction.

Parse into working sets:
  VALID_NOTES = set of all note stems (notes[*].stem)
  VALID_MOCS  = set of all MOC stems  (mocs[*].stem)
  VALID_EXTERNAL_MOCS = MOC stems referenced in cross-domain links
                        that exist in other vaults. Always valid;
                        never flag as broken.

Also read and store per-note metadata for overlap detection:
  For each note: stem, title, description, domain, moc, status

If validation_errors[] is non-empty in the index:
  Pre-quarantine those stems — do not process them as READY notes.
  Report them in Phase 5 under "Quarantined (index-flagged)".

Add every READY note's stem from this batch to VALID_NOTES now,
so batch notes are valid wikilink targets for each other.

Do not re-read VAULT_INDEX.json again during this run.

════════════════════════════════════════════════════════════════
PHASE 1 — CLASSIFY AND PARSE (one pass over all inputs)
════════════════════════════════════════════════════════════════
Classify each note in 00_Inbox/Note Pipe/:

  READY    — has Gist / Evidence / Debate / Application structure
             (NotebookLM four-header format)
             OR has both "Concept:" and "Why it matters:" fields
  FRAGMENT — flying note missing both required fields
  SKIP     — source file, PDF link, or non-concept content

For FRAGMENT and SKIP: log to report only. Do not process.

For each READY note, extract and store in memory:
  title_raw, icon, core, mechanism, application, source, ideas, debate

Do not build any files yet.

INPUT MAPPING RULES

Three input formats are supported. Try them in order; the first that
matches the block's headers wins.

(1) PRIMARY — structured-block format (NotebookLM, per
    99_System/EXTRACT_PROMPT_NotebookLM.md). Identified by the
    presence of labelled `Concept title:` and `Stem (safe):` fields
    at the top of the block.

  Block frontmatter labels → concept-note frontmatter (mostly verbatim):
    Concept title          → title
    Stem (safe)            → safe_stem CANDIDATE (re-validated per
                              Phase 3 stem rules; do not trust blindly)
    Icon                   → icon
    Aliases                → aliases (must end with ≥2; reject if fewer)
    Description            → description (apply YAML single-quoting per
                              Phase 3 rule when value contains " : [[ **)
    Suggested MOC          → moc: candidate. Validate against VALID_MOCS:
                                - exact match in VALID_MOCS → accept
                                - "NEW: <name>" → flag as MOC CANDIDATE
                                  in Phase 5 report, do NOT create the
                                  MOC (still requires human confirm)
                                - "[UNKNOWN]" → STOP and ASK the user
                                  before Phase 3 writes the note
                                - any other unknown → treat as UNKNOWN
    Suggested cluster      → annotation used by Phase 4 as the candidate
                              cluster-insertion point; not stored in the
                              concept note itself
    Tier                   → annotation only; affects Phase 4 cluster
                              placement:
                                atomic           → insert under suggested
                                                   cluster
                                meta-organising  → flag for placement at
                                                   cluster-overview tier
                                                   (target of instance-of
                                                   edges from siblings)
    Possible overlap       → seed for Phase 2 overlap detection. If
                              non-NONE, run overlap check against the
                              named stem FIRST before broader scan.
    Tags                   → frontmatter tags (split the comma-separated
                              list into a YAML list)

  Block body labels → concept-note sections (verbatim — the prompt is
  designed so the block body is already in the final shape):
    ## Core Idea (Summary)             → ## Core Idea (Summary)
    **Example:**                       → keep inline under Core Idea
    **Metaphor:**                      → keep inline under Core Idea
                                          (flashcard extractor regex
                                          requires this exact spelling)
    ## Mechanism (Key details)         → ## Mechanism (Key details)
    **Boundary condition:**            → keep as last bulleted item in
                                          Mechanism (flashcard extractor
                                          requires this exact spelling)
    ## Application                     → ## Application
    **When to use:** / **When NOT...** → keep as labelled bullets
    **Counter-instance:**              → if non-NONE, keep as an extra
                                          bullet under Application; if
                                          [NONE], omit the line
    ## Debate                          → not a standalone section in the
                                          final note. If the Debate text
                                          references another extracted
                                          concept, that target goes into
                                          the Links block's Contradicts
                                          line. Otherwise discard the
                                          Debate text (its value was the
                                          edge it implied).
    ## Typed edges                     → populate the Links block with
                                          one line per edge label:
                                          - supports          → Supports:
                                          - contradicts       → Contradicts:
                                          - prerequisite-for  → Prerequisite-for:
                                          - instance-of       → Instance-of:
                                          - mechanism-of      → Mechanism-of:
                                          - mitigates         → Mitigates:
                                          - related-to        → Related:
                                          Multiple targets on the same
                                          label go on one line, comma-
                                          separated, in encounter order.
                                          Quote and confidence are
                                          carried into Phase 4 audit
                                          memory (used to populate the
                                          MOC's Concept Relationships
                                          subsections) but NOT into the
                                          concept note's Links block.
    ## Source - Filename / Location    → Links Source [[filename]] +
                                          source_type frontmatter
                                          (inferred from extension:
                                          .pdf → book, .epub → book,
                                          otherwise → article unless
                                          obvious). Location is stored
                                          as a comment after the Source
                                          line:  `<!-- src-loc: ... -->`

  [SOURCE CONFLICT: ...] or [APPLICATION UNCLEAR ...] markers anywhere
  in the block → flag in Phase 5 report; do not silently strip them.

(2) LEGACY — Gist / Evidence / Debate / Application / Relationships
    structure (older NotebookLM extractions). Use as fallback if a
    block lacks the structured-block labels:

  Gist          → Core Idea (Summary)
  Evidence      → Mechanism bullets (Boundary condition = last bullet,
                                     re-derived from narrative)
  Debate        → Contradicts wikilink (not a standalone section)
  Application   → When to use / When NOT to use
  Relationships → Parsed in Phase 4 to populate MOC Concept Relationships
                  section. Only "SUPPORTS" and "CONTRADICTS" appear in
                  legacy extractions; map them straight through.
                  Format: [A] SUPPORTS [B] → "[[A_underscore]] supports
                  [[B_underscore]]"
                  Format: [A] CONTRADICTS [B] → "[[A_underscore]]
                  contradicts [[B_underscore]]"
                  Convert all concept titles to underscore filename
                  form before writing. Only include edges where both
                  endpoints will exist in the same MOC.

(3) LEGACY — Concept / Why it matters structure (flying note):
  Concept        → Core Idea (Summary)
  Why it matters → first Mechanism bullet
  Agent infers remaining Mechanism bullets and Boundary condition

════════════════════════════════════════════════════════════════
PHASE 2 — AUDIT ALL NOTES (one table, not per note)
════════════════════════════════════════════════════════════════
Output a single markdown table before building anything:

| Note | Safe chars | Feynman fail | Unflagged acronym | Domain | No boundary | No when-not | Overlap | Split |

Rules:

  Safe chars    — title contains em-dash, accents, underscores, or
                  chars outside [A-Za-z0-9 ' \-().] → auto-fix in Phase 3

  Feynman fail  — Core Idea uses unexplained jargon — domain terms
                  without inline definition. For the deterministic
                  acronym check, see "Unflagged acronym" below.
                  → auto-fix in Phase 3

  Unflagged     — Body prose (Core Idea + Mechanism + Application) contains
   acronym       a token matching \b[A-Z]{2,}\b that is NONE of:
                    (a) on the allowlist at 99_System/acronym_allowlist.md
                    (b) expanded inline within ±80 chars (parenthetical
                        whose word-initials spell the acronym, in either
                        direction, case-insensitive)
                    (c) a stem, title, or alias in VAULT_INDEX
                        (notes + patterns + decisions + solutions + ideas)
                    (d) the note's own title or alias (self-reference)
                  Tokens inside fenced code blocks or inline backticks
                  are excluded from the scan.
                  → STOP. Present the human with a per-acronym resolution
                  menu before Phase 3:
                    1. expand inline           (rewrite the sentence)
                    2. wikilink to vault       (replace with [[stem]])
                    3. flag as vault gap       (write plain text with
                                                "(not yet in vault: ...)")
                    4. add to allowlist        (append to 99_System/
                                                acronym_allowlist.md)

  Domain        — Shows the proposed `domain:` value derived from the
                  note's moc: assignment. Resolution path:
                    (a) Single MOC in moc:[] → inherit domain from
                        that MOC's frontmatter (auto-fix in Phase 3)
                    (b) Multiple MOCs across different domain values
                        → set domain: multiple (auto-fix in Phase 3)
                    (c) moc: [] (orphan note) → STOP and ask the user
                        to pick a domain before Phase 3
                    (d) MOC's domain is not in VALID_DOMAINS (legacy
                        value, typo, or unmapped) → STOP and ask
                        whether to use multiple, fix the source MOC,
                        or extend VALID_DOMAINS
                  Column values shown in audit table: the inherited
                  string (e.g. "strategy") for (a)/(b); "ASK: orphan"
                  for (c); "ASK: MOC domain '{value}' invalid" for (d).
                  → Auto-fix for (a) and (b); STOP for (c) and (d).

  No boundary   — Mechanism section lacks "Boundary condition:" bullet
                  → auto-fix in Phase 3

  No when-not   — Application section lacks "When NOT to use:"
                  → auto-fix in Phase 3

  Overlap       — Title semantically overlaps with an existing VALID_NOTES
                  entry. Classify as one of:
                    TYPE 1 (true duplicate — same mechanism, same scope)
                    TYPE 2 (satellite — same mechanism, narrower/domain scope)
                    TYPE 3 (adjacent — related but distinct mechanism)
                    MERGE  (richer instance — new note has better mechanics,
                            patch existing note instead of creating new one)
                  → STOP and wait for human confirmation before resolving

  Split         — Note conflates two distinct mechanisms (different
                  boundary conditions)
                  → STOP and wait for human confirmation

Auto-fix: Safe chars, Feynman fail, No boundary, No when-not, Domain (inheritance paths a/b)
STOP for: Overlap, Split, Unflagged acronym, Domain (orphan moc:[] or invalid MOC domain)

OVERLAP DETECTION RULES
Check every new note title against VALID_NOTES for:
  1. Exact title match → TYPE 1 (true duplicate)
  2. Spelling variant (same words, different capitalisation or spelling)
     → TYPE 1, treat as duplicate
  3. Same mechanism described in a narrower domain context
     → TYPE 2 (satellite), propose rename: {Concept} in {Domain}
  4. Same phenomenon, richer or different mechanics from different source
     → MERGE, patch existing note with new mechanics, discard new title
  5. Related vocabulary, genuinely different boundary condition
     → TYPE 3, keep both, add Supports/Contradicts link

Also check within the batch itself — if the same concept appears
twice in the same batch (from the same or different sources),
detect it before writing either instance.

════════════════════════════════════════════════════════════════
PHASE 3 — BUILD ALL NOTES (one pass, validate before each write)
════════════════════════════════════════════════════════════════

SAFE CHAR RULES — two separate sets: TITLE and STEM

  TITLE (the title: frontmatter field — human-readable)
    Keep  : A-Za-z0-9 space ' - ( ) . , & "
    Strip accents: é→e, è→e, à→a, ü→u, ö→o, ñ→n, ç→c
    Replace em-dash (—) → " - ", en-dash (–) → " - "
    Strip : leading/trailing whitespace
    Example: "Conway's Law Isomorphism" ✓
    Example: "R&D-to-Sales Efficacy Ratio" ✓

  STEM (the filename without .md — must match [[wikilinks]])
    Start from title, then apply in order:
      1. Strip possessive 's  : Conway's → Conway
      2. Strip any remaining ' : can't → cant
      3. Replace vs.          : vs. → vs   (period after abbreviation)
      4. Replace Mr. Dr. etc. : Mr. → Mr   (same rule — no periods in stems)
      5. Replace spaces       : " " → "_"
      6. Strip unsafe chars   : remove " [ ] & , / \ : * ? < > | .
         (period stripped everywhere except in source filenames like .pdf)
      7. Collapse __          : re.sub(r'_+', '_', s)
      8. Strip leading/trailing underscores
    Keep  : A-Za-z0-9 _ - ( )
    Examples:
      "Conway's Law Isomorphism"           → Conway_Law_Isomorphism
      "R&D-to-Sales Efficacy Ratio"        → RD-to-Sales_Efficacy_Ratio
      "Convince vs. Persuade"              → Convince_vs_Persuade
      "Mr. Market as an Emotional Party"   → Mr_Market_as_an_Emotional_Party
      "The \"Enough\" Threshold"           → The_Enough_Threshold
      "Volatility as a Fee, Not a Fine"    → Volatility_as_a_Fee_Not_a_Fine

  Source file references (.pdf, .md, .epub) are EXEMPT from stem rules.
  Use the exact filename: [[on-writing-well.pdf]]

  Result: stem must equal filename without .md, character for character.

FEYNMAN FIX RULES (auto-applied before writing — acronyms are NOT
auto-fixed; the Phase 2 "Unflagged acronym" column STOPS the build
and requires human resolution per-acronym)
  If Core Idea contains a technical term a non-specialist would not
  understand → replace with a plain-language equivalent or add an
  inline parenthetical definition
  If Core Idea is a topic label rather than an assertive claim
  (contains "is about", "involves", "refers to") → rewrite as a
  falsifiable assertion

DOMAIN ASSIGNMENT RULE (applied at write time)
  The agent NEVER picks a domain by guessing from the content. Domain
  is derived from the note's moc: assignment by the following rule:

    (a) moc: [SingleMOC]              → inherit domain from
                                        SingleMOC's frontmatter
    (b) moc: [MOC_A, MOC_B] where      → set domain: multiple
        their domain values differ
    (c) moc: [MOC_A, MOC_B] where      → inherit that single shared
        their domain values match      domain value
    (d) moc: []                        → STOP. Ask the user to pick
                                        a domain from VALID_DOMAINS
                                        OR to assign a MOC first.
                                        Never default.
    (e) MOC's domain is not in         → STOP. Ask whether to use
        VALID_DOMAINS                   `multiple`, fix the source
                                        MOC, or extend VALID_DOMAINS.

  The Phase 2 audit "Domain" column shows the inferred value for (a),
  (b), (c) and an "ASK:" string for (d), (e). Phase 3 only writes the
  domain after audit clears.

  Rationale: every domain mistake we've ever seen was the agent
  inventing a value not in VALID_DOMAINS (`'Enterprise Architecture'`,
  `'Knowledge Management'`, etc.) by free-association from the content.
  Inheriting from the MOC makes the controlled vocabulary the source
  of truth and forces invalid values to be fixed at the MOC level,
  not silently propagated across every note that joins the MOC.

WIKILINK RULE
  Obsidian resolves wikilinks by FILENAME, not by title.
  All wikilink text must match the filename stem exactly
  (filename without .md extension). Filenames use underscores
  as word separators — so ALL wikilinks must use underscores:
    CORRECT:   [[Focused_Mode]]
    INCORRECT: [[Focused Mode]]
    CORRECT:   [[Cortisol_-_The_Stress_Hormone]]
    CORRECT:   [[MOC_Learning_and_Memory]]

  Exceptions that do NOT use underscores:
    - Source file references: [[on-writing-well.pdf]]
    - Empty links: [[]]

  A [[link]] is valid if it exists in:
    VALID_NOTES  — filename stems of vault concept notes
    VALID_MOCS   — filename stems of vault MOC files
    VALID_EXTERNAL_MOCS — MOC stems from other domains (always valid)
  If the target does not yet exist as a file, write:
    (not yet in vault: "Title")
  Never write a broken [[wikilink]].

OVERLAP RESOLUTION (applied after human confirmation for flagged notes)
  TYPE 1 / spelling variant → skip entirely. Log as SKIPPED.
  TYPE 2 / satellite → rename to "{Concept} in {Domain}". Write with
    Related: [[parent concept note]] in Links.
    Assign to the domain MOC, not the parent concept's MOC.
  TYPE 3 / adjacent → write both. Add Supports or Contradicts link
    between them. Assign to the same MOC cluster.
  MERGE → do NOT write a new file. Patch the existing note:
    Insert the new mechanics into the existing Mechanism section.
    Add the new source to the Source field if different.
    Log as MERGED in report.

SATELLITE NAMING RULE
  {safe_title(original concept)} in {Domain}
  Examples:
    "Pyramid Principle in Architecture Writing"
    "Active Recall in Architecture Reviews"
  The domain must be the specific domain context of the new note,
  not a repeat of the parent concept's domain.

ALIAS RULE
  alias_1 = short colloquial name or acronym
  alias_2 = alternative framing of the mechanism
  Both must contain only safe title characters (apostrophes allowed).

RENAME PROTOCOL (applies whenever safe_stem changes a note's filename)
  Renaming is an atomic operation — partial propagation is worse
  than no rename. Execute all four steps before moving to the next note:
  1. Write the file with the new stem as filename.
  2. Immediately scan EVERY file in VALID_NOTES + VALID_MOCS for
     [[old_stem]] and replace with [[new_stem]].
  3. Update the old_stem entry in VALID_NOTES to new_stem so
     subsequent notes in the same batch use the correct link.
  4. Log the rename and all propagation targets in the run report.
  Never proceed to the next note until propagation is complete.

validate_note() — hard gate before every file write
  ✓ Frontmatter parses as valid YAML — run yaml.safe_load on the
    block between --- delimiters before any other check.
    If this fails → fix inline, re-validate once, then QUARANTINE.
  ✓ description: field uses single-quoted YAML string if it contains
    any of: "  :  [[  **
    Format:  description: 'value with "quotes" or colons: here'
    Internal single quotes doubled: description: 'Conway''s Law'
    Apply this quoting rule at write time, not at review time.
  ✓ filename stem == safe_stem(title)  (character for character)
  ✓ aliases count ≥ 2, each containing only safe title characters
  ✓ type, status, description, domain, source_type, created,
    updated, moc, reviewed, flashcards fields all present and non-empty
  ✓ domain is one of: learning | writing | presentations |
    biology | strategy | multiple | enterprise-architecture | ai-systems |
    data-engineering
  ✓ source_type is one of: book | course | article |
    notebooklm | shower-thought
  ✓ status is one of: draft | needs-review | final
  ✓ moc: is a YAML list — e.g. [MOC_Learning_and_Memory]
  ✓ reviewed and flashcards are boolean (true or false, not string)
  ✓ Links block has the 4 REQUIRED lines in this order:
      - Related: ...
      - Contradicts: ...
      - Supports: ...
      - Source [[...]]
    Optionally, between Supports: and Source, the block MAY contain any
    subset of these typed-edge lines (in this order if present):
      - Prerequisite-for: ...
      - Mechanism-of: ...
      - Instance-of: ...
      - Mitigates: ...
    Any other label is rejected. The 4 required lines are never reordered
    or omitted (use [[]] if the field has no value).
    (MoC: line removed — moc: frontmatter is the single source of truth)
  ✓ Mechanism contains "**Boundary condition:**"
  ✓ Application contains "**When NOT to use:**"
  ✓ No [[wikilink]] contains spaces — all wikilinks must use
    underscores matching the exact filename stem
  ✓ Every [[wikilink]] text exists in VALID_NOTES, VALID_MOCS,
    or VALID_EXTERNAL_MOCS (external MOC refs are always valid)

  On failure: fix inline → re-validate once.
  Still failing: write to 00_Inbox/QUARANTINE/ and log in report.
  Never write an invalid note to the vault.

PHASE 4 — UPDATE ALL MOCs (one pass, all patches together)
════════════════════════════════════════════════════════════════
Collect all MOC insertions needed across the entire batch.
Group by MOC file. Apply all insertions to a given MOC in
one patch operation — not one per note.

MOC FIT ASSESSMENT
  To determine which MOC a note belongs to, use the index first —
  do not open MOC files just to assess fit.
  $idx.mocs contains description: and stem: for every MOC.
  Read those descriptions to determine fit.
  Only open a MOC file if the index description is ambiguous and
  you need to read the ## Core Question section. In that case use
  Read with limit:40 to avoid loading the SVG and cluster lists.

CLUSTER ASSIGNMENT RULE
  A note belongs to a cluster if its mechanism answers the same
  sub-question the cluster addresses — not because it shares
  vocabulary with the cluster label.
  If the mechanism fits an existing cluster → insert there.
  If the mechanism does not fit any existing cluster but fits
  the MOC's core question → propose a new cluster name to the
  human before inserting.
  If the mechanism does not fit the MOC's core question at all
  → flag as SATELLITE or MOC CANDIDATE (see below).

MOC UPDATE RULES
  1. Insert new note into the correct cluster with format:
       [[Note_Stem]] — one sentence on how it answers the core question
     Note: wikilinks in MOC cluster lists use stems, not titles.
  2. Update note_count in MOC frontmatter after every batch.
  3. Check Open Questions after every batch — if any new note
     answers an open question, update that question entry:
       Before: "No note on X — gap"
       After:  "[[Note_Stem]] now covers X. Remaining gap: Y"
  4. Add cross-domain links if new notes explicitly connect to
     concepts in other MOCs (e.g. via Contradicts or Supports links).
  5. Patch only — never rewrite the full MOC file.
     If insertion point is ambiguous, flag for human — do not guess.

MOC WIKILINK VALIDATION (runs on every MOC patch before writing)
  After building the patch text, scan every [[wikilink]] in the
  patched section:
  - Any link containing spaces → convert to underscore stem form
  - Any link not in VALID_NOTES or VALID_MOCS → replace with
    plain text: (not yet in vault: "Title")
  - Never write a MOC patch that contains a broken wikilink.
  This check is part of the same Phase 4 write operation —
  not a separate pass.

MOC CANDIDATE RULE
  < 4 similar notes with no MOC → flag as SATELLITE (nearest MOC)
  ≥ 4 similar notes with no MOC → flag as MOC CANDIDATE
  Do not create new MOC files. Flag only and wait for confirmation.

KEY TENSIONS AND OPEN QUESTIONS
  For each updated MOC, check if Key Tensions should be revised.
  Propose the change — do not write without human confirmation.
  Open Questions: update automatically when a new note answers one.
  Do not ask for confirmation on Open Question updates.

CLUSTER MAP GENERATION — moved to visualise-moc
  Phase 4 NO LONGER generates the ## Cluster Map section. That work is
  now owned by the separate `visualise-moc {MOC_stem}` command, which
  renders a two-tier Mermaid diagram (overview + cluster detail) and
  patches the MOC's `### Cluster Map` subsection with transclusions
  pointing to the rendered diagram file.

  See docs/visualisation-pipeline.md for the full spec.

  Phase 4 still does NOT touch the `### Cluster Map` subsection at all.
  The user (or the visualise-moc command) is responsible for
  regenerating diagrams when clusters change. The note_count and
  last_cluster_review frontmatter fields are still updated by Phase 4
  itself (note_count from this batch's additions; last_cluster_review
  is left to visualise-moc to update).

  CANONICAL MOC SECTION ORDER:
    ## Core Question

CONCEPT RELATIONSHIPS AGGREGATION
  During Phase 3 Build, the agent already has every new note's typed
  Links lines in memory: Related (→ related-to), Contradicts, Supports,
  Prerequisite-for, Mechanism-of, Instance-of, Mitigates. In Phase 4,
  for each MOC being patched:

  1. Collect all typed-edge links from the new notes being added to
     this MOC, across the 7 closed-vocab labels.
  2. Append at the END of the MOC file, after ## Cross-Domain Links,
     under a ## Concept Relationships header. Emit ONE subsection per
     edge type that has at least one edge, in this order (omit empty
     subsections entirely):
       ## Concept Relationships
       ### Supports
       - [[Note A]] supports [[Note B]]

       ### Contradicts
       - [[Note C]] contradicts [[Note D]]

       ### Prerequisite-for
       - [[Note E]] prerequisite-for [[Note F]]

       ### Mechanism-of
       - [[Note G]] mechanism-of [[Note H]]

       ### Instance-of
       - [[Note I]] instance-of [[Note J]]

       ### Mitigates
       - [[Note K]] mitigates [[Note L]]

       ### Related-to
       - [[Note M]] related-to [[Note N]]
  3. Only include edges where both endpoints exist in the same
     MOC. Cross-MOC edges go to "## Cross-Domain Links" instead.
  4. Deduplicate: if [[A]] {label} [[B]] already appears under the
     matching subsection, do not add it again.
  5. The aggregation is additive — never delete existing edges.
     Stale edges are removed only during the weekly health check.

  These 7 subsections are the single source of truth for the diagram
  generator (visualise-moc). Keeping them in the MOC means the diagram
  can be rendered from one file without scanning every concept note.
